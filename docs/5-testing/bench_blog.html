What is being measured

The bench harness uses a seeded RNG so keypairs are identical across runs. Payloads are pre-generated once. No allocation happens inside the timing loop.

// Deterministic keypair — same across every run
let rng = ChaCha20Rng::from_seed([0x42u8; 32]);
let signing_key = EcdsaP256Signer::generate(rng);

// Payload pre-generated outside timing loop
let payload: Vec<u8> = (0..size).map(|i| i as u8).collect();

// Measured: DSSE PAE + sign + bundle JSON
c.bench_function("sign_blob/1kb", |b| {
    b.iter(|| sign_blob(&signing_key, &payload, "application/octet-stream"))
});

The Signer SPI takes &self (not &mut self), enabling Arc<dyn Signer> sharing. The bench exercises EcdsaP256Signer — the default. Ed25519, P-384, and secp256k1 are feature-gated implementations of the same trait.

The 200× gap

sign_blob at 234 µs vs cosign sign-blob at ≥ 50 ms. The gap is not algorithmic — both use P-256 ECDSA on the same DSSE-encoded bytes. The gap is architectural.

Why verify is slower than sign for small payloads

At 1 KB, sign_blob takes 234 µs but verify_blob takes 1.07 ms — verify is 4.6× slower for the same payload size.

sign (1 KB)
234 µs
verify (1 KB)
1.07 ms
sign (1 MB)
4.94 ms
verify (1 MB)
2.30 ms

P-256 ECDSA verification requires two scalar multiplications (one for the public key, one for the signature point). Signing requires one. For small payloads, SHA-256 hashing is negligible and the scalar multiplications dominate — verify loses.

At 1 MB the order reverses: SHA-256 hashing of the PAE bytes (proportional to payload) dominates and the verify path has cheaper post-hash dispatch. Verify at 1 MB reaches 434 MiB/s vs sign's 202 MiB/s.

This is not a justsign implementation choice — it is a property of ECDSA arithmetic on any P-256 implementation.

Throughput scaling

At large payloads, SHA-256 throughput (the PAE encoding hashes the full payload) becomes the bottleneck. This is an x86 SHA-NI hardware ceiling — not a justsign ceiling.

The keyless caveat

The bench tests the static-key path — the lower bound on signing latency. The keyless path (sign_blob_keyless) adds:

StepTypical costjustsign vs cosign?
Fulcio HTTPS round-trip~100–300 msIdentical — same network operation
Rekor submission~100–200 msIdentical — same network operation
The 200× advantage applies to the cryptographic signing operation itself. Keyless flows are dominated by network latency, which is the same whether you use justsign or cosign. The advantage is most pronounced in high-throughput batch pipelines that pre-fetch OIDC tokens and submit to Rekor in bulk.

vs cosign

Run scripts/bench/compare_cosign.sh on Linux or WSL2 (cosign requires Linux for the static-key flow). The script generates a throwaway P-256 key and times cosign sign-blob --key for each payload size.

The cosign numbers include Go subprocess startup (~50 ms floor). The comparison targets the common-ancestor case — P-256 static key — where both tools perform the same cryptographic work. The only difference is in-process vs subprocess.

cosign is not a bad tool. It is a subprocess designed for interactive use, not for embedding in signing loops. The 50 ms floor is not negligence — it is a property of what a Go subprocess is. justsign is the library answer to the same problem.

Reproducing

Prerequisites

RequirementNotes
Rust stable toolchainrequired
cosign (comparison only)Linux/WSL2; GitHub release or brew install cosign

Steps

1. Clone and enter the workspace

git clone git@github.com:sweengineeringlabs/justsign.git
cd justsign

2. Run the benchmark

cargo bench -p swe_justsign_sign --bench sign_verify

3. Run a single case

cargo bench -p swe_justsign_sign --bench sign_verify -- "sign_blob/1kb"
cargo bench -p swe_justsign_sign --bench sign_verify -- "verify_blob/1mb"

4. Compare against cosign (WSL2/Linux)

bash scripts/bench/compare_cosign.sh

Output

Criterion prints to stdout. HTML reports are written to:

target/criterion/sign_blob/report/index.html
target/criterion/verify_blob/report/index.html

These files are ephemeral — cargo clean removes them.
